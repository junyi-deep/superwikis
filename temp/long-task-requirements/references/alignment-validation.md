# 对齐验证执行协议

## 何时运行

专家级轨道步骤 E10。由 SKILL.md 在分类/编写/验证/粒度/推迟（E9）之后、SRS 审查者（E11）之前调用。

## 目的

向后验证：给定已写的 SRS，它实际上是否解决了 E1 中确定的根本问题？此步骤捕获最危险的需求失败形式 — 正式正确但解决错误问题的 SRS。

这是内部检查。除非具体发现需要，否则没有用户交互。

## E10a. 根因可追溯性检查

对于疼痛图中的**每个**行（来自 E1，存储在第 1.3 节）：

1. 找到至少一个 FR，其 EARS 语句或验收标准解决此疼痛点
2. 如果疼痛点没有解决的 FR → 检查它是否出现在第 1.2 节不在范围内，有明确排除理由
3. 如果疼痛点既未解决也未排除 → 这是**可追溯性差距**

对于 5-为什么根本原因（来自 E1）：

1. 验证至少一个 FR 直接解决根本原因（不仅仅是症状）
2. 如果根本原因未解决 → 标记为可追溯性差距

**关于可追溯性差距**：
- 1-2 个差距：通过创建新的最小 FR 或添加明确的范围外条目来自动解决（选择需要最少新获取的那个）
- 3+ 个差距：通过 `AskUserQuestion` 呈现差距表，询问哪些差距应成为新 FR vs 明确排除

## E10b. JTBD 结果验证

定位来自 E1 的 JTBD 声明（存储在第 1.3 节）。

检查："如果用户完成第 4 节中的每个必须优先级 FR，是否实现了 JTBD '所以我可以 [结果]'？"

- **是** → 继续
- **否** → 识别 JTBD 结果中未覆盖的部分。通过 AskUserQuestion 向用户呈现：
  > "您声明的目标是'[JTBD 结果]'。当前需求没有完全覆盖 [缺失方面]。我应该为此添加需求，还是当前范围足够？"
  - 用户要添加 → 创建新 FR，返回到 E9 进行分类
  - 用户确认当前范围足够 → 记录为 **PARTIAL**，继续

**门控**：JTBD 验证在解决前阻止 E11。可接受的结果：
- **PASS** — JTBD 完全可实现
- **PARTIAL** — 用户明确确认当前范围足够，尽管 JTBD 覆盖不完整

FAIL（在用户确认前 JTBD 差距）不能继续到 E11。

## E10c. 事前分析

LLM 自我评估（除非发现不平凡，否则没有用户交互）：

> "如果我们按书面构建 SRS 中的所有内容，什么仍然会让用户不满意？"

对照以下检查：
- 来自 E2 的变通方案探测答案 — 当前变通方案中的每个令人沮丧的步骤是否都有至少一个 FR 解决？
- 来自 E3 的场景演练叙述 — 所有提取的流程差距是否都在最终 FR 列表中解决？
- 来自 E5 的隐藏需求 — 每个"是"答案是否都成为明确的 NFR？
- 疼痛图项目 — 是否有任何只是部分解决（变通方案消除但根本原因仍然存在）？

对于每个事前分析发现：
- 应该是 FR → 添加它（返回到 E9 进行分类）
- 应该是 NFR → 添加它
- 已知风险但现在不可操作 → 添加到第 11 节开放问题

## E10d. 孤立 FR 检测（镀金检查）

对于第 4 节中的每个 FR，检查它是否有可追溯来源：
- 链接到疼痛图行（解决已声明的疼痛点）
- 链接到 JTBD 结果（实现声明目标所需）
- 来源于演练步骤（E3 提取）
- 来源于隐藏需求（E5）

如果 FR **没有**来自上述任何来源的可追溯来源：
- 检查是否有其他 FR 依赖它（基础设施/实用 FR 通常没有直接疼痛点链接是必要的）
- 如果没有依赖存在 → 在第 11 节开放问题中标记：
  > "FR-xxx 没有可追溯的疼痛点或 JTBD 链接 — 确认在范围内或推迟到未来增量。"

**不要**自动删除孤立 FR。向用户展示是行动。

## 输出

将对齐验证结果写入 SRS 第 1.3 节：

```
**对齐验证**：PASS / PARTIAL / FAIL
- 根本原因覆盖：M 个疼痛点中解决了 N 个
- JTBD 结果：已实现 / 部分实现（用户确认）/ 未实现
- 事前分析发现：添加了 N 项 / 发现 0 项
- 孤立 FR 标记：开放问题中有 N 项 / 0 项
```

**PARTIAL**（用户确认 JTBD）是可接受的，不阻止 E11。
JTBD 上 FAIL（E10b）且没有用户确认，在解决前阻止 E11。
