# 假设修正执行协议

## 何时运行

专家级轨道步骤 E4。由 SKILL.md 在场景演练（E3）之后调用。

## 目的

填写每个 FR 的边界、错误处理、权限和数据生命周期 — 而无需提出开放式问题。LLM **主动推导**具体行为假设，并呈现给用户确认/修正，而不是让用户从零开始想象边缘案例。

## 核心机制

对于每个 FR（或 2-3 个相关 FR 组），LLM 从所有可用上下文（演练叙述、领域知识、疼痛图、步骤 1 上下文）**推导具体行为假设**。这些作为结构化的**行为假设表**呈现给用户确认（✓）、修正（✗）或补充（+）。

## 行为假设表格式

通过 AskUserQuestion 呈现：

> "根据您的演练，我假设 **[FR 标题]** 的以下内容。请标记每行：✓ 正确、✗ 错误（告诉我正确的）、+ 添加我遗漏的内容。"

| # | 维度 | 我的假设 | ✓/✗/+ |
|---|---|---|---|
| 1 | **快乐路径** | 当 [触发器]，系统 [动作]，导致 [结果] | |
| 2 | **无效输入** | 如果 [特定无效输入]，系统 [假设错误行为，例如显示内联错误、静默拒绝、自动纠正] | |
| 3 | **边界** | 最大 [N 项 / 文件大小 / 字符] 是 [假设限制]；超出 [假设行为] | |
| 4 | **失败路径** | 如果 [外部依赖] 不可用，系统 [假设后备：重试 / 排队 / 显示错误 / 优雅降级] | |
| 5 | **权限** | 只有 [假设角色] 可以执行此操作；未授权用户看到 [假设行为：403 / 隐藏按钮 / 重定向] | |
| 6 | **并发** | 如果两个用户同时 [动作]，[假设冲突解决：最后写入优先 / 锁 / 合并 / 排队] | |
| 7 | **数据生命周期** | 此操作创建的数据 [永久保留 / X 后过期 / 用户可删除 / 管理员可归档] | |
| 8 | **负面范围** | 此 FR **不** [假设排除 — 用户可能期望但不在范围内的内容] | |

**每个假设必须具体且明确**，而不是泛泛的。坏的："如果无效输入，系统显示错误。" 好的："如果电子邮件字段不包含 @ 符号，系统在该字段下方显示内联错误并阻止提交。"

## FR 类型定义与维度选择

并非所有 8 个维度都适用于每个 FR。根据 FR 类型选择：

| FR 类型 | 定义 | 包含维度 | 跳过维度 |
|---|---|---|---|
| **只读显示** | 向用户显示数据而不修改状态 | 1, 3, 4, 5 | 2（无输入）、6（无写入）、7（无数据创建） |
| **数据录入 / 表单** | 用户提交新数据或编辑现有数据 | 1, 2, 3, 4, 5, 7 | 6（除非多用户编辑可能） |
| **状态改变动作** | 触发状态转换（批准、取消、发布、删除） | 1, 2, 4, 5, 6, 7 | 3（除非批处理） |
| **后台进程** | 无直接用户交互运行（cron 作业、队列消费者、同步） | 1, 4, 7 | 2, 3, 5, 6（无直接用户交互） |
| **集成 / API** | 从外部系统发送或接收数据 | 1, 3, 4, 8 | 5（除非用户面向）、6, 7 |

**始终包含维度 8（负面范围）** 对于边界模糊的复杂 FR（范围内和范围外之间）。

## 批处理规则

- 当 2-3 个密切相关的 FR 共享工作流或数据模型时，按 AskUserQuestion 分组
- 如果单个 FR 需要所有 8 个维度，单独呈现
- 每轮更喜欢更少 FR 更多维度 — 每轮深度优于广度

## 收敛机制

假设修正协议是**自动终止**的，没有硬性上限：

- ✓（已确认）→ 假设正确，无需进一步行动
- ✗（已修正）→ LLM 用用户的修正更新 FR
- \+（已添加）→ 用户添加 LLM 遗漏的维度 → 新验收标准（或如果描述不同能力，则是新 FR）

**当所有 FR 中没有新的 ✗ 或 + 修正浮现时，假设修正完成。** 没有开放式跟进问题，没有"还有什么？"探测 — 表结构按设计穷尽行为维度。

如果 ✗ 修正揭示重大新复杂性（例如，"实际上，根据金额有三种不同的批准工作流"），将修正视为新子 FR 的触发器，并为新发现的子 FR 运行另一个假设表。

## 每行输出

| 用户标记 | 成为 |
|---|---|
| ✓ 已确认快乐路径 | FR 的 EARS 语句 |
| ✓ 或 ✗ 边界/错误/权限/并发/生命周期 | Given/When/Then 验收标准 |
| \+ 添加维度 | FR 的新验收标准，或如果描述不同能力则是新候选 FR |
| ✓ 已确认维度 8（负面范围） | EXC-xxx 排除条目 |

## 质量检查

所有假设表完成后，验证：
- 每个 FR 至少有 一个错误/边界验收标准（满足现有 A6 反模式检查）
- 每个有外部依赖的 FR 都有指定的失败路径
- 每个有用户输入的 FR 都有指定的无效输入处理
