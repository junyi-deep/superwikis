# 问题框架执行协议

## 何时运行

专家级轨道步骤 E1。由 SKILL.md 调用。**不要**在轻量级轨道运行。

## E1a. 从上下文推导（内部，无用户交互）

在提问之前，从用户的初始描述推导：
- 起草一句话问题陈述
- 列出如果系统建成其生活将改变的参与者
- 假设 2-3 个当前状态的疼痛点
- 标记用户是否陈述了**解决方案**（"为我构建 X"）而不是**问题**（"我不能做 Y"）— 解决方案锚定的请求必须在 E1b 中质疑

## E1b. 问题框架轮（单个 AskUserQuestion，≤4 个问题）

根据项目上下文调整措辞。跳过其答案已从步骤 1 上下文明确无歧义的任何问题。

1. **5-为什么种子**："没有这个系统，今天会出现什么问题？带我走过一个最近真实的例子 — 发生了什么，你不得不做什么代替，以及代价是什么（时间、金钱、错误、沮丧）？"
   - 从答案内部推导 WHY 链（最多 3 层）。在用户答案支持的最深层原因处停止。**不要**发明超出陈述或明确暗示的原因。

2. **JTBD 探测**："你最终想要实现什么结果 — 不是你想要的工具，而是你的世界中的成功是什么样子的？"
   - 目标答案形式："我想要 [动机]，这样我可以 [结果]"
   - 如果答案仍然是解决方案形式（"我想要一个仪表板"），跟进一次："拥有它让你能做什么是你今天不能做的？"

3. **疼痛排名**："在你描述的问题中，哪个让你付出最多代价 — 而且多久发生一次？（每天 / 每周 / 每月）"
   - 接受定性排名；**不要**强制用户给出他们没有的数字。

4. **解决方案挑战**（*仅在用户在原始描述中提出了具体解决方案时询问*）："这是解决这个问题的唯一方法，还是如果你能以其他方式实现 [JTBD 结果] 你也会满意？"
   - 目的：检测用户是否锚定在特定实现上，而更简单的解决方案可以满足 JTBD。
   - 如果用户**没有**提出具体解决方案，完全跳过此问题。

## E1c. 构建产出（内部，无进一步用户问题）

从答案中产生三个将嵌入 SRS 第 1.3 节的产出：

**5-为什么链**：
```
症状：[用户用他们自己的话陈述的问题]
为什么 1：[第一个原因]
为什么 2：[为什么 1 的原因]
为什么 3：[最深的支持原因 — 除非答案更进一步，否则在此停止]
根本原因：[系统原因，需求必须解决]
```
在用户答案支持的最深 WHY 处停止。标记停止点。

**JTBD 声明**：
```
当 [情况]，我想要 [动机]，这样我可以 [结果]。
```
从用户的 JTBD 探测答案构建。尽可能使用他们自己的话。

**疼痛图**：
| 疼痛点 | 当前变通方案 | 频率 | 严重性 (高/中/低) | 分数 (F×S) |
|---|---|---|---|---|
| [疼痛 1] | [他们今天做什么] | 每天/每周/每月 | 高/中/低 | 3/2/1 × 3/2/1 |

分数：频率（每天=3，每周=2，每月=1）× 严重性（高=3，中=2，低=1）。最高分数 = 最高优先级解决的疼痛。

## E1d. 内部检查点

在继续 E2 之前，验证：你能陈述系统**为什么**需要存在、**谁**最受当前状态之苦，以及**什么**是构成成功的最小结果 — 不猜测吗？

- **是** → 继续到 E2
- **否** → 再提出一个针对性的 AskUserQuestion 来解决差距

## 输出

所有三个产出 → 写入 SRS 第 1.3 节（问题陈述）。
这些喂养：
- E2（变通方案探测使用疼痛图）
- E3（从疼痛图推导的演练工作流）
- E10（从疼痛图和 JTBD 向后检查的对齐验证）
