# 场景演练执行协议

## 何时运行

专家级轨道步骤 E3。由 SKILL.md 在问题框架（E1）和扩展范围（E2）之后调用。

## 目的

在深入单个 FR 之前，发现宏级别流程差距、集成点和隐含排序。逐能力提问获取会遗漏跨能力需求 — 演练通过强迫用户叙述完整旅程来暴露它们。

## 需要多少演练

每个主要工作流一次演练。从疼痛图顶部项目和 E2 答案确定主要工作流。

| 专家级规模 | 演练数量 |
|---|---|
| 小（5–15 FR） | 1–2 |
| 中（15–50 FR） | 2–3 |
| 大（50–200+ FR） | 3–5（每个 epic/域一个） |

## 演练提示（每个工作流一个 AskUserQuestion）

> "带我走过一个完整的 [工作流名称]，从头到尾。从你决定 [触发器] 的那一刻开始。告诉我你采取的每个步骤，系统在每一点应该显示或做什么，你在哪里停止。包括当事情出错时会发生什么。"

根据特定工作流调整触发器。例如：
- "...从你需要入职新员工的那一刻..."
- "...从客户下订单的那一刻..."
- "...从你注意到数据中异常的那一刻..."

## LLM 提取（内部，无用户交互）

对于每个叙述答案，提取并分类：

| 类别 | 要找什么 | 示例 | 行动 |
|---|---|---|---|
| **显性步骤** | 系统行为的直接陈述 | "我点击'提交'，系统保存记录" | → 候选 FR |
| **隐含步骤** | 用户略过或认为"显而易见"的动作 | "然后我会登录"（隐含身份验证） | → 候选 FR，标记待确认 |
| **流程差距** | 用户跳过而没有解释机制的转换 | "然后报告出现"（如何？手动触发？自动生成？计划的？） | → 有针对性的跟进问题 |
| **集成点** | 外部系统、数据源、向其他工具的交接 | "我将其导出到 Salesforce" | → 候选 IFR |
| **错误提及** | 对失败、异常或恢复的任何引用 | "有时 API 超时" | → 相关 FR 的候选验收标准 |

## 流程差距跟进（每个工作流单个 AskUserQuestion，如果有差距）

如果提取产生了需要确认的流程差距或隐含步骤，在一个跟进中询问：

> "在你 [工作流] 的演练中，我注意到几点我想确认：
> 1. 在 [步骤 A] 和 [步骤 B] 之间，[转换] 如何发生 — 用户触发它，还是系统自动执行？
> 2. 你提到 [模糊步骤] — 在那一点屏幕上应该具体显示什么？
> 3. 当 [错误提及] 发生时，预期的恢复路径是什么？"

根据差距数量调整问题数量。如果没有发现差距，完全跳过跟进。

## 输出

- **候选 FR 列表**，包含来源（哪个演练、哪个步骤、显性 vs 隐含）
- **候选 IFR 列表**，来自集成点
- **候选验收标准**，来自错误提及

所有输出喂养假设修正（E4）— 每个候选 FR 在下一步中获得行为假设表。

## 收敛

此步骤有界限：
- 主要工作流数量（在演练开始前确定，有限）
- 用户叙述长度（有限 — 用户在故事结束时停止）
- 流程差距跟进数量（有限，提取列表确定）

没有开放式"还有什么？"或"我遗漏了什么？"问题。演练结构本身在其范围内是穷尽的。
