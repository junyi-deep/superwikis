## 规则

- **首先发现问题** — 在任何裁决之前，跨所有维度的 5+ 个项目（步骤 1 不是可选的）
- **应用所有检查** — 即使期望通过，也不要跳过任何组
- 要具体 — 引用确切的需求 ID、节号或图表元素
- **不要**审查实现选择或设计决策 — SRS 指定 WHAT，不是 HOW
- 裁决由评分标准计算 — 您不能用自己的叙述解释覆盖 NO
- 每个问题一个顾虑 — 不要将多个失败捆绑在一个问题编号下
- **模糊词总是 R2/A1 违规** — "快速"、"容易"、"健壮"没有数字阈值 = 失败，无例外
- **复合需求总是失败 R3** — 如果单个陈述可以拆分为两个独立的通过/失败测试，它必须被拆分
- **占位符图表 = D1 或 D3 FAIL** — 包含仅有 `%%` 注释或模板占位符文本的 Mermaid 代码块不计入图表
- **IFR 节（第 6 节）豁免 A3** — 接口需求合法使用技术术语（REST、JSON、HTTP）
- **有理由的"[Not applicable]"可以接受** — 任何节都将 S2 检查标记为 YES，如果所有缺失节都明确标记并有解释
- **仅当 SRS 有零个用户面向 FR 时才跳过 D 检查** — 如果任何 FR 涉及用户交互，图表是强制性的
- **G 检查仅适用于第 4 节 FR** — 待办列表文档中的推迟项豁免粒度检查
- **G3 的"故意粗粒度"理由可以接受** — 如果 FR 明确指出其多个标准是同一行为的变体，则标记 G3 为 YES

## 问题分类启发式

使用这些规则为步骤 1-2 中的每个问题分配 `解决类型`。

**始终 USER-INPUT**（永远不要自动修复 — 需要领域知识）：
- 用作需求但未量化的质量属性："快速"、"可扩展"、"可靠"、"容易"、"直观"，没有数字阈值 → 询问实际指标（R2/A1）
- TBD/TBC/占位符文本在需求内容中 → 询问实际值（A5）
- 范围外决策：什么排除 vs 推迟是业务决策 → 询问用户（C5）
- 冲突必须级别优先级：只有用户可以调和优先级争端（R5）
- 缺少利益相关方可追溯性：只有用户可以确认需求服务于哪个利益相关方需求（R1）

**通常是 USER-INPUT**（除非从获取上下文清楚，否则分类为 USER-INPUT）：
- 涉及业务规则的错误/边界案例缺失（A6、C1）— 例如，"支付失败时会发生什么？"需要用户输入；当无效电子邮件提交时会发生什么？"通常可以推断
- "正确行为"由业务领域知识定义的模糊验收标准

**始终 LLM-FIXABLE**（结构/句法 — 无需领域知识）：
- 复合需求拆分（A2）：在"和"/"或"上机械拆分为独立需求
- 设计泄露重写（A3）：使用现有上下文将实现词汇重写为可观察行为
- 无代理被动语态（A4）：添加"系统将"或命名代理
- 缺少唯一 ID（R8）：从 SRS 中建立的序列分配
- 节结构：自动填充缺失节为"[Not applicable]"（S2-S4）
- 可追溯性矩阵：从需求 ID 列表自动填充（S3）
- 图表生成（D1-D4）：从 SRS 中现有的参与者和 FR 列表生成
- NFR 测量方法添加（C3）：仅当阈值已由用户指定时添加"通过 [标准工具] 测量"
- 单个 FR 中多个参与者（G1）：按参与者机械拆分 — 每个参与者的不同动作成为单独 FR
- CRUD 捆绑（G2）：机械拆分为单独操作（创建、读取、更新、删除）作为单独 FR

**通常是粒度的 USER-INPUT**（除非从上下文明显，否则分类为 USER-INPUT）：
- 场景爆炸（G3）：当 FR 有 4+ 验收标准覆盖不同路径时，询问用户哪些场景真正独立 vs 哪些是同一行为的变体

**永远不要发明领域值**：
如果 SRS 说"快速"且在获取过程中没有给出阈值，**唯一正确的解决类型是 USER-INPUT** — 不是发明"200ms"。
