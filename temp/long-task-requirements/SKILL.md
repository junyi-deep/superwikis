# 需求获取与 SRS 生成

通过系统化的获取、质疑和验证流程，将原始想法转化为结构化的高质量软件需求规范（SRS）文档——符合 ISO/IEC/IEEE 29148 和 EARS 需求语法标准。

自动适应深度：**轻量级轨道**适用于范围清晰的项目（3-5 轮交互），**专家级轨道**适用于复杂领域（10-20 轮交互）。两者产生相同的 SRS 模板输出。

<HARD-GATE>
在您展示 SRS 文档并获得用户批准之前，**不要**调用任何设计技能、实现技能、编写任何代码、搭建任何项目，或采取任何设计/实现行动。这适用于**每个项目**，无论其看似多么简单。
</HARD-GATE>

## 反模式："这太简单了，不需要 SRS"

每个项目都要经过此流程。待办事项列表、单一功能工具、配置更改——全都是。"简单"项目正是未经验证的假设造成最多浪费工作的地方。SRS 可以很短（对于真正简单的项目，几句话就够了），但您**必须**展示它并获得批准。

## 检查清单

您**必须**为以下每个项目创建一个 TodoWrite 任务，并按顺序完成：

1. **探索项目上下文** — 读取现有文档、代码、约束；检测 SRS 模板
2. **复杂度评估** — 评估 5 个信号，选择轻量级或专家级轨道（内部进行）
3. **问题与范围获取**
   - 轻量级：L1（聚焦问题与范围，1 轮）
   - 专家级：E1（通过 `references/problem-framing.md` 进行问题框架）+ E2（扩展范围）
4. **功能需求获取**
   - 轻量级：L2（扁平能力轮次）
   - 专家级：E3（通过 `references/scenario-walkthrough.md` 进行场景演练）+ E4（通过 `references/hypothesis-correction.md` 进行假设修正）
5. **NFR + 隐藏需求**
   - 轻量级：L3（合并，1 轮）
   - 专家级：E5（隐藏需求）+ E6（NFR 量化）
6. **约束、假设、术语表** — 两个轨道相同
7. **需求分类** — 功能性 / NFR / 约束 / 假设 / 接口 / 排除
8. **编写需求** — 应用 EARS 模板，分配 ID，编写验收标准，生成图表
9. **验证 SRS** — 检查 8 个质量属性，检测反模式，验证可测试性
10. **粒度分析** — 通过 6 个启发式方法（G1-G6）检测过大的 FR，分解候选项，用户批准非平凡拆分
11. **范围适配与推迟** — 评估当前轮次与下一轮次，如有需要生成推迟待办列表
12. **【仅专家级】对齐验证** — 通过 `references/alignment-validation.md`
13. **SRS 合规审查** — 派遣 srs-reviewer 子代理；门控：所有检查通过才能继续
14. **展示与批准 SRS** — 轻量级：单一合并步骤；专家级：逐节展示
15. **保存 SRS 和待办列表** — `docs/plans/YYYY-MM-DD-<topic>-srs.md` + 推迟待办列表（如有）并提交
16. **过渡到 UCD** — **必需的子技能：** 调用 `long-task:long-task-ucd`

**最终状态是调用 long-task-ucd。** 不要调用任何其他技能。

## 步骤 1：探索上下文

1. 彻底阅读用户提供需求文档 / 想法描述
2. 探索项目将基于或集成现有代码 / 代码库
3. 识别初始约束：技术栈、平台、集成、法规
4. 检查 SRS 模板：
   - 如果用户指定了模板路径 → 读取并验证它
   - 否则 → 读取此技能附带的默认模板 `docs/templates/srs-template.md`
   - **验证**：模板必须是包含至少一个 `## ` 标题的 `.md` 文件

## 步骤 1.5：复杂度评估（内部进行 — 无用户交互）

在步骤 1 之后，根据用户描述和项目上下文评估 5 个复杂度信号：

| # | 信号 | 轻量级指标 | 专家级指标 |
|---|---|---|---|
| S1 | **声明的范围** | 单一日的，清晰边界（"执行 X 的脚本"） | 模糊/广泛范围（"用于管理...的平台"、"一个...系统"） |
| S2 | **参与者数量** | 1 个参与者或无用户面向交互 | 提及 2+ 个不同用户角色 |
| S3 | **集成表面** | 无外部系统，或 1 个知名 API | 2+ 个外部系统、自定义协议 |
| S4 | **领域复杂度** | 开发者工具、工具、充分理解的领域 | 具有术语的业务领域、监管暴露、多利益相关方 |
| S5 | **描述风格** | 解决方案特定（"使用 Y 构建 X"） | 问题导向或模糊（"我们需要更好的 X"、"用户抱怨 Y"） |

**≥3 个专家级信号 → 专家级轨道。否则 → 轻量级轨道。**

在内部记录层级决策。**不要**询问用户使用哪个层级。

### 升级触发器（轻量级 → 专家级）

在轻量级获取过程中，如果出现**任何**以下情况，无缝切换到专家级轨道：
- 2+ 个具有冲突需求的不同用户角色
- 浮现 3+ 个外部系统集成
- 提及监管/合规要求
- 用户与自己的早期答案矛盾（信号不清晰的心智模型）
- 获取后 FR 数量超过 10

关于升级：到目前为止收集的所有轻量级产出都成为专家级输入。**不要**重新开始或宣布破坏性切换 — 只需开始提出更深入的问题（E1 问题框架、E3 演练等）。

## 轻量级轨道

适用于范围清晰、单一参与者、充分理解领域的项目。目标：3-5 轮交互。

### L1：聚焦问题与范围（单个 AskUserQuestion，≤4 个问题）

1. "这个系统解决什么问题，当它正常工作时会是什么样子？"
2. "谁使用它，在什么环境（桌面/移动/CLI/API）？"
3. "这个版本明确排除什么？"
4. "有什么硬约束 — 语言、平台、托管、许可证？"

输出：SRS 第 1 节的一句话问题陈述、参与者列表、范围边界、约束。

如果 Q1 的答案模糊或问题导向 → 升级触发器启动 → 切换到专家级。

### L2：扁平能力获取（1-3 轮，每轮 ≤4 个问题）

对于每个能力区域，按轮次提问（最多 4 个问题）：
- 用户做什么？（触发/动作）
- 系统如何响应？（可观察行为）
- 什么输入是无效的，应该发生什么？
- 确认一个具体的 Given/When/Then 示例

当相关能力共享工作流时，将它们分组在同一轮中。跨多轮拆分大型能力区域。

### L3：快速 NFR + 隐藏需求检查（单个 AskUserQuestion）

1. "有什么性能目标 — 响应时间、吞吐量、数据量？"
2. "这是否处理个人数据、面临法规或需要无障碍支持？（如果是的，哪些？）"
3. "多种语言或时区？"
4. "有什么超出基本身份验证的安全要求吗？"

对 Q2 的任何"是" → 内联生成 EARS 格式的 NFR 候选。如果 Q2 揭示重大监管暴露 → 升级触发器。

### L4–L6：分类、编写、验证、展示、保存

轻量级获取后，继续**共享步骤**（检查清单中的步骤 7-16）：
- L4 = 步骤 7-8（分类，EARS，图表）
- L5 = 步骤 9-11 + 步骤 13（验证、粒度、推迟，SRS 审查者 Group P = PASS-SKIPPED）
- L6 = 步骤 14-16（将整个 SRS 作为单一批准块展示，保存，过渡到 UCD）

## 专家级轨道

适用于具有复杂领域、多个参与者或范围不清晰的项目。目标：10-20 轮交互。

### E1：问题框架【仅专家级】

阅读 `references/problem-framing.md` 并严格遵循。

**摘要**：单个 AskUserQuestion（≤4 个问题）— 5-为什么种子、JTBD 探测、疼痛排名、解决方案挑战。产出：5-为什么链、JTBD 声明、疼痛图 → 嵌入 SRS 第 1.3 节。

### E2：扩展范围轮【仅专家级】

使用 E1 答案释放的槽位。单个 AskUserQuestion（≤4 个问题）。用有针对性的探测替换已在 E1 中回答的标准第 1 轮问题：

- **变通方案探测**："带我走过 [疼痛图中] 最烦人的步骤。是什么让它令人沮丧 — 是手动的、容易出错的、缓慢的还是不透明的？"
  → 用户在当前变通方案中讨厌的每个步骤都是一个候选 FR。

- **环境探测**："这通常在哪里何时完成 — 在有大屏幕的桌面上、在现场的移动设备上、在时间压力下，还是在团队之间共享？"
  → 揭示 UX、离线、移动优先、多用户和无障碍约束。

加上尚未被 E1 回答的剩余标准范围问题（不在范围内、约束）。

**规则**：问题总数 ≤4。优先考虑能揭示新信息的探测，而不是重新询问 E1 已经覆盖的内容。

### E3：场景演练【仅专家级】

阅读 `references/scenario-walkthrough.md` 并严格遵循。

**摘要**：每个主要工作流一次演练（1-3 个工作流）。用户叙述端到端。LLM 提取显性步骤、隐性步骤、流程缺口、集成点和错误提及。针对流程缺口进行跟进（有界限，由提取数量决定）。

### E4：假设修正【仅专家级】

阅读 `references/hypothesis-correction.md` 并严格遵循。

**摘要**：针对每个 FR（或 2-3 个相关 FR 组），展示行为假设表，其中包含适用维度（由 FR 类型选择）。用户标记 ✓/✗/+。当没有新的修正浮现时自然收敛。

### E5：隐藏需求【仅专家级】

单个 AskUserQuestion，复选框式（是/否 + 告诉我更多），≤4 个探测：

1. **监管/合规**："这个系统是否处理受监管的数据或流程？（个人数据 → GDPR/CCPA；健康 → HIPAA；支付 → PCI-DSS；金融 → SOX；政府 → 特定行业）"
   - 是 → 隐含 NFR：数据驻留、审计日志、违规通知时间线、同意管理、数据保留限制

2. **无障碍**："任何用户是否有无障碍需求 — 视力障碍、运动限制、老年人、屏幕阅读器用户或仅键盘导航？这个会在移动设备上运行吗？"
   - 是 → 隐含 NFR：WCAG 2.1 AA 合规、键盘可导航性、最小触摸目标（44×44px）、足够颜色对比度（4.5:1）

3. **隐私设计**："系统会收集、存储或处理个人身份信息（姓名、电子邮件、位置、行为数据、设备 ID）吗？"
   - 是 → 隐含 NFR：数据最小化、用户可控的数据导出/删除、同意记录、违规响应时间

4. **国际化**："任何用户会使用 [检测到的主要语言] 以外的语言交互，或来自不同的时区或地区吗？"
   - 是 → 隐含 NFR：区域感知日期/时间/货币格式、字符串外部化（无硬编码 UI 文本）、RTL 布局（如适用）、时区感知存储

**规则**：任何"是" → 在继续之前以 EARS 格式创建 NFR 候选。标记来源 = "隐藏（E5）"。E6 量化阈值。

**智能跳过**：如果步骤 1 上下文清楚显示为纯内部、无 PII、单语言、非监管的开发者工具 → 将所有四个探测合并为一个确认：
> "这似乎是一个内部工具，没有个人数据、没有受监管行业暴露、没有无障碍要求，也没有 i18n 需求 — 正确吗？"

### E6–E8：NFR、约束、术语表

与标准获取相同的结构：

**E6（NFR 量化）**：使用与当前第 N+1 轮相同的探测。将 E5 候选吸收为预填充行 — 量化其阈值。

| 类别（ISO 25010） | 探测 |
|---|---|
| **性能** | 响应时间目标？吞吐量？并发用户？ |
| **可靠性** | 可用性目标？恢复时间？数据丢失容忍度？ |
| **可用性** | 无障碍要求？可学习性标准？ |
| **安全性** | 身份验证方法？授权模型？数据加密？ |
| **可维护性** | 模块化约束？测试覆盖率目标？ |
| **可移植性** | 平台限制？浏览器支持？ |
| **可扩展性** | 当前负载？目标负载？增长时间线？ |

跳过明显不相关的类别。**规则**：每个 NFR 必须有**可衡量标准**。

**E7（约束与接口）**：硬限制、假设、外部系统契约。

**E8（术语表）**：具有潜在歧义的领域术语。

### E9：分类、编写、验证、粒度、推迟

与检查清单中的共享步骤 7-11 相同。与标准流程没有差异。

### E10：对齐验证【仅专家级】

阅读 `references/alignment-validation.md` 并严格遵循。

**摘要**：根因可追溯性（疼痛图 → FR 覆盖）、JTBD 结果验证（**门控 — 失败时阻止 E11**）、事前分析、孤立 FR 检测。输出 → SRS 第 1.3 节对齐验证字段。

### E11：SRS 审查者、展示、保存

与共享步骤 13-16 相同，有两个差异：
- SRS 审查者包括 **Group P**（活跃的，不是 PASS-SKIPPED）
- 对于非平凡项目逐节展示（不是单一合并批准）

## 步骤 7-11：共享质量管道（两个轨道）

### 步骤 7：分类需求

组织为类别：

| 类别 | ID 前缀 | 描述 |
|---|---|---|
| 功能性 | FR-001 | 可观察的系统行为 |
| 非功能性 | NFR-001 | 具有可衡量标准的质量属性 |
| 约束 | CON-001 | 限制解决方案空间的硬限制 |
| 假设 | ASM-001 | 假定为真的信念；记录失效风险 |
| 接口 | IFR-001 | 外部系统契约 |
| 排除 | EXC-001 | 明确排除在范围内 |

### 步骤 8：使用 EARS 模板编写需求

将 EARS 模板应用于每个功能需求：

| 模式 | 模板 | 何时使用 |
|---|---|---|
| **普遍性** | 系统应 `<action>`。 | 始终开启的行为 |
| **事件驱动** | 当 `<trigger>` 时，系统应 `<action>`。 | 对事件的响应 |
| **状态驱动** | 当 `<state>` 时，系统应 `<action>`。 | 行为取决于模式/状态 |
| **不期望行为** | 如果 `<condition>`，则系统应 `<action>`。 | 错误处理、容错 |
| **可选** | 在 `<feature/config>` 的地方，系统应 `<action>`。 | 可配置能力 |

对于每个需求，还要编写：
- **验收标准** — 至少一个具体的 Given/When/Then 场景
- **视觉输出**（如果面向 UI）— 描述用户看到什么变化的一句话（渲染意图，不是渲染规范）。例如："蛇在游戏板上每次滴答后的位置在视觉上更新。"对于没有用户可见输出的 FR 写"N/A — 仅后端"。此字段为下游阶段（特性设计、TDD）提供明确的视觉语言来推导渲染契约。
- **优先级** — 必须 / 应该 / 可以 / 不会（MoSCoW）
- **来源** — 可追溯到哪个利益相关方需求或用户故事

#### 8c. 生成图表

在所有需求编写完成后，生成可视化辅助：

**用例视图**（第 3.1 节）：`graph LR`，所有参与者为 `Actor((Name))`，所有 FR-xxx 作为 `subgraph System Boundary` 内的用例节点，参与者到用例的参与为有向边。

**流程**（第 4.1 节）：每个功能区域一个 `flowchart TD`，其中有 3+ 个顺序步骤或分支。开始/结束为 `([label])`，决策为 `{condition?}`，带有 `-- YES -->` / `-- NO -->` 标签。

### 步骤 9：验证 SRS 质量

#### 9a. 每个需求检查（8 个质量属性）

| # | 属性 | 检查 | 红旗 |
|---|---|---|---|
| 1 | **正确** | 可追溯到已确认的利益相关方需求？ | 孤立需求（镀金） |
| 2 | **无歧义** | 两个读者会写出相同的测试用例？ | 模糊词："快速"、"健壮"、"用户友好" |
| 3 | **完整** | 所有输入、输出、错误案例、边界都定义了？ | "包括但不限于..." |
| 4 | **一致** | 不与其他需求矛盾？ | 时间冲突、格式冲突 |
| 5 | **排序** | 有 MoSCoW 优先级？ | 一切都是"高优先级" |
| 6 | **可验证** | 可以编写通过/失败测试？ | "系统应该易于使用" |
| 7 | **可修改** | 在确切的一个地方陈述？ | 跨节重复 |
| 8 | **可追溯** | 有唯一 ID + 来源链接？ | 缺少 ID 或孤立 |

#### 9b. 反模式检测

| 反模式 | 检测信号 | 修复 |
|---|---|---|
| **模糊形容词** | "快速"、"大型"、"可扩展"没有数字 | 量化 |
| **复合需求** | "和" / "或" 连接两个独立能力 | 拆分 |
| **设计泄露** | "类"、"表"、"端点" | 重写为行为 |
| **无代理被动语态** | "数据将被验证" — 由谁？ | 添加代理 |
| **TBD / TBC** | 未解决的占位符 | 解决或开放问题 |
| **缺少否定** | 只指定了正向案例 | 添加错误/边界案例 |
| **不可测试 NFR** | NFR 没有可衡量阈值 | 添加指标 |

#### 9c. 完整性交叉检查

- 每个功能区域至少有一个错误/边界案例
- 所有外部接口都有数据格式 + 协议
- 所有 NFR 都有测量方法，不仅仅是目标
- 术语表涵盖所有领域特定术语
- 不在范围内部分列出推迟的功能

### 步骤 10：粒度分析与分解

将 6 个启发式方法应用于每个 FR。如果**任何**一个触发，该 FR 是分解候选：

| # | 启发式方法 | 检测信号 |
|---|---|---|
| G1 | **多个参与者** | 2+ 个不同角色执行不同动作 |
| G2 | **CRUD 捆绑** | 创建 + 读取 + 更新 + 删除作为单一需求 |
| G3 | **场景爆炸** | 4+ 个验收标准覆盖不同路径 |
| G4 | **跨层关注** | 后端逻辑和用户面向 UI 在一个 FR 中 |
| G5 | **多状态行为** | 3+ 个不同系统状态或模式 |
| G6 | **时间耦合** | 触发事件 + 延迟/计划后果 |

对于分解候选：识别原子行为，应用单一责任测试，保持可追溯性（FR-003 → FR-003a，FR-003b），重新验证子项。

| 候选数量 | 行动 |
|---|---|
| 0 | 跳过 |
| 1-3 | 自动分解；内联展示理由 |
| 4+ | 通过 AskUserQuestion 展示给用户批准 |

### 步骤 11：范围适配与推迟

评估是否所有需求都属于当前轮次。应用范围适配标准（优先级、依赖性、完整性、风险、范围预算）。向用户展示推迟建议。如果推迟获得批准，生成 `docs/plans/YYYY-MM-DD-<topic>-deferred.md`。

规则：
- 必须优先级的 FR **永远不**自动推迟
- 依赖完整性 — 如果 FR-X 依赖 FR-Y，两者都保留
- 推迟待办列表保留 EARS + 验收标准以便增量拾取

## 步骤 13：SRS 合规审查

派遣子代理独立验证 SRS：

```
Task(
  subagent_type="general-purpose",
  prompt="""
  您是符合 ISO/IEC/IEEE 29148 的 SRS 合规审查者。
  阅读审查者提示：skills/long-task-requirements/prompts/srs-reviewer-prompt.md

  项目上下文：
  {project_context}

  完整 SRS 草稿（所有节）：
  {srs_draft}

  需求 ID 列表：
  {requirement_id_list}

  按照提示严格进行审查。
  """
)
```

**所有检查必须通过才能继续：**
- Group R (R1-R8)：质量属性
- Group A (A1-A6)：反模式
- Group C (C1-C5)：完整性
- Group S (S1-S4)：结构合规
- Group D (D1-D4)：图表
- Group G (G1-G3)：粒度
- Group P (P1-P4)：问题对齐（专家级轨道；轻量级轨道为 PASS-SKIPPED）

**失败时双轨解决：**

**轨道 1：USER-INPUT 项 → 立即询问**

使用 `AskUserQuestion` 发送有针对性的问卷 — **不要**转储完整审查报告。

**轨道 2：LLM-FIXABLE 项 → 自动修复**

并行修复所有 LLM-FIXABLE 项。重新派遣审查者（第 2 轮）。

**最大：2 轮重新派遣。** 第 2 轮失败后 → 升级给用户。

## 步骤 14-16：展示、保存、过渡

### 步骤 14：展示与批准 SRS

- **轻量级轨道**：将整个 SRS 作为一个块展示。单一批准步骤。
- **专家级轨道（< 5 FR）**：合并批准步骤。
- **专家级轨道（≥ 5 FR）**：逐节展示：
  1. 目的、范围、问题陈述与排除
  2. 术语表与用户画像
  3. 功能需求
  4. 非功能需求
  5. 约束、假设与接口

展示每节。等待用户反馈。在继续下一节之前纳入更改。

### 步骤 15：保存 SRS 文档与推迟待办列表

保存到 `docs/plans/YYYY-MM-DD-<topic>-srs.md`。

阅读步骤 1 中找到的模板：
1. 保持模板的标题结构
2. 用批准的 SRS 内容替换每个标题下的指导文本
3. 如果还没有，添加顶部元数据（`Date`、`Status`、`Standard`、`Template` 路径）
4. 对于未覆盖的模板节：标记"[Not applicable]"
5. 对于没有匹配模板节的批准内容：附加为"Additional Notes"

如果在步骤 11 生成了推迟待办列表，随同保存：`docs/plans/YYYY-MM-DD-<topic>-deferred.md`。提交两者。

## 扩展表

| 层级 | 信号 | 典型 FR 数量 | 获取深度 | 批准 |
|---|---|---|---|---|
| **轻量级** | <3 个专家级信号 | 1-10 | L1-L3（扁平轮次，合并 NFR） | 合并单一步骤 |
| **专家级（小）** | ≥3 个专家级信号 | 5-15 | E1-E5（1-2 次演练，分组假设） | 2-3 节 |
| **专家级（中）** | ≥3 个专家级信号 | 15-50 | E1-E5（2-3 次演练，按 FR 假设） | 逐节 |
| **专家级（大）** | ≥3 个专家级信号 | 50-200+ | E1-E5（3-5 次演练，批处理假设） | 逐节 |

## 红旗

| 合理化 | 正确回应 |
|---|---|
| "这太简单了，不需要 SRS" | 轻量级轨道就是简单路径。它在 3-5 轮中产生一个简短的 SRS。 |
| "用户已经描述了他们想要什么" | 用户描述是原始输入；SRS 添加结构、完整性、可测试性 |
| "我可以在设计期间弄清楚需求" | 需求定义 WHAT；在 HOW 期间发现它们会导致返工 |
| "NFR 不适用于这个项目" | 每个项目至少都有隐含的性能/可靠性需求 — 让它们变得明确 |
| "术语表是显而易见的" | 对谁来说是显而易见的？定义每个用户和开发者可能不同理解的术语 |
| "我只从快乐路径开始" | 错误案例、边界和否定必须现在捕获 |
| "这个 FR 作为一个大需求很好" | 应用 6 个粒度启发式方法（G1-G6）— 隐藏复杂性创建超大特性 |
| "所有需求都属于这一轮" | 范围适配评估确保专注 — 推迟低优先级项 |
| "推迟的项目可以放在不在范围内" | 不在范围内是散文；推迟待办列表保留 EARS + 验收标准 |
| "这很复杂但我会用轻量级来节省时间" | 复杂度评估存在是有原因的。如果 ≥3 个专家级信号触发，使用专家级。 |
| "跳过演练，我有足够的 FR" | 演练发现每 FR 提问遗漏的跨能力差距 |
| "假设表有太多维度" | 按 FR 类型选择维度（只读 5 行，数据录入 7 行）。不是全部 8 个。 |
| "无障碍不适用于这个项目" | 任何用户界面都有无障碍影响。WCAG 2.1 AA 是底线。 |
| "我们会在设计中处理 GDPR/隐私" | 隐私要求必须在 SRS 中。数据模型和同意流程依赖于它们。 |
| "专家路径轮次太多，跳过一些步骤" | 每个专家级步骤防止下游返工。如果项目真的更简单，应该是轻量级。 |
