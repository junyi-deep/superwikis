# 设计文档生成

将批准 SRS 作为输入。提出实现方法，获得逐节设计批准，并产出回答 HOW 的设计文档 — 而 SRS 回答 WHAT。

<HARD-GATE>
在您展示设计且用户批准之前，**不要**调用任何实现技能、编写任何代码、搭建任何项目、运行 init_project.py，或采取任何实现行动。这适用于**每个项目**，无论其看似多么简单。
</HARD-GATE>

## 反模式："SRS 已经足够详细，可以开始编码"

SRS 描述系统**必须做什么**。设计文档描述**如何做**。即使需求非常清晰，实现方法（架构、数据模型、技术栈选择）也需要明确决策和用户批准。跳过设计会导致会话中期更正和返工。

## 检查清单

您**必须**为以下每个项目创建一个 TodoWrite 任务，并按顺序完成：

1. **阅读批准 SRS** — 从 `docs/plans/*-srs.md`
2. **探索技术上下文** — 现有代码、框架、部署环境
3. **提出 2-3 种方法** — 带有权衡和您的建议
4. **逐节设计批准** — 架构、数据模型、API、UI、测试、部署
5. **编写设计文档** — 保存到 `docs/plans/YYYY-MM-DD-<topic>-design.md` 并提交
6. **过渡到 ATS** — **必需的子技能：** 调用 `long-task:long-task-ats`

**最终状态是调用 long-task-ats。** 不要调用任何其他实现技能。

## 步骤 1：阅读 SRS 和 UCD 并提取设计输入

1. 从 `docs/plans/*-srs.md` 阅读批准 SRS 文档
2. 从 `docs/plans/*-ucd.md` 阅读批准 UCD 风格指南（如果存在 — 仅对 UI 项目）
3. 提取关键设计驱动因素：
   - **功能范围** — FR 数量、优先级分布、依赖链
   - **NFR 阈值** — 影响架构的性能目标、可靠性、可扩展性
   - **约束** — 限制技术/方法选择的硬性限制
   - **接口需求** — 要集成的外部系统、协议、数据格式
   - **用户画像** — 影响 API/UI 设计决策的技术水平
   - **UCD 风格标记**（如果 UCD 存在）— 调色板、排版、间距、组件目录 → 为前端架构和 UI/UX 节提供信息
4. 列出任何必须在设计继续之前解决的 SRS **开放问题**
   - 如果未解决的问题影响架构 → 在步骤 2 之前通过 `AskUserQuestion` 询问用户

## 步骤 2：探索技术上下文

1. 探索项目将基于的现有代码/代码库
2. 识别 SRS 中没有的技术约束（例如，单体仓库结构、CI/CD 管道、现有库）
3. 检查设计文档模板：
   - 如果用户指定了模板路径 → 读取并验证它
   - 否则 → 读取 `docs/templates/design-template.md`（此技能附带的默认模板）
   - **验证**：模板必须是包含至少一个 `## ` 标题的 `.md` 文件

## 步骤 3：提出方法

展示**2-3 种实现方法**，带有明确权衡：

```markdown
## 方法 A：[名称]
**如何工作**：[1-2 句话]
**优点**：[要点列表]
**缺点**：[要点列表]
**最佳时机**：[条件]
**NFR 影响**：[此方法如何影响 SRS NFR 阈值]
**第三方依赖**：[此方法需要的关键库/框架，带版本]

## 方法 B：[名称]
...

## 建议：方法 [X]
**理由**：[为什么这最适合 SRS 约束和 NFR]
```

**关键**：每种方法必须根据 SRS 约束和 NFR 阈值进行评估。不能达到"必须" NFR 的方法被取消资格。

## 步骤 4：逐节批准

对于非平凡项目，将设计分成几节并逐节获得批准：

1. **架构** — 系统组件、逻辑视图、技术栈决策
   - 必须包含**逻辑视图**（Mermaid `graph`），显示层/包/模块和依赖方向
   - 必须包含**组件图**（Mermaid `graph`），显示运行时组件和交互
   - 必须根据 SRS 约束证明技术栈选择的合理性
   - 必须显示如何满足 NFR 阈值
2. **关键特性设计** — 每个关键特性或特性组一章
   - 每个特性章节**必须**至少包含：
     - **类图**（Mermaid `classDiagram`）— 类/模块、属性、方法、关系
     - **一个行为图**：序列图（Mermaid `sequenceDiagram`）或流程图（Mermaid `flowchart`）
     - **集成表面**（§4.N.6）— 声明带 §6.2 契约 ID 的 Provides/Requires 表；如果没有跨特性依赖写"Self-contained"
   - 对于复杂特性，包含所有四种视图：类图、序列图、流程图和设计笔记
   - 所有图表**必须**使用 **Mermaid** 格式 — 不使用 ASCII 艺术，不使用图片引用
3. **数据模型** — 模式、关系、存储策略
   - 必须使用 Mermaid ER 图（`erDiagram`）（如适用）
4. **API / 接口设计**
   - **外部接口**（§6.1）— 端点、契约、协议（追溯到 SRS IFR-xxx）
   - **内部 API 契约**（§6.2）— 特性到特性边界；每个 §3.3 组件图边必须有对应的 §6.2 行，带契约 ID、请求/响应模式和错误代码
5. **UI/UX 方法**（如适用）— 布局策略、交互模式
   - 必须解决 SRS 用户画像
   - 如果 UCD 文档存在：必须引用 UCD 风格标记（颜色、排版、间距）和组件目录
   - 前端架构决策（组件库、状态管理、路由）必须与 UCD 风格标记兼容
   - 包含映射：UCD 组件提示 → 具体实现组件
6. **第三方依赖** — 所有带**确切版本号**的库/框架
   - 必须验证依赖之间的相互兼容性
   - 必须验证与项目目标运行时版本的兼容性
   - 必须记录每个依赖的许可证类型
   - 必须包含非平凡依赖链的依赖图（Mermaid）
7. **测试策略** — 仅高级测试方法决策
   - 测试哲学：带质量门控的 TDD（Red → Green → Refactor → Coverage → Mutation）
   - 工具选择：测试框架、覆盖率工具、突变工具（带版本 — 这些是设计决策）
   - 覆盖率阈值：line >= X%、branch >= Y%、mutation >= Z%
   - **边界**："详细的 requirement-to-test-category 映射、NFR 测试方法和跨特性集成场景在 ATS 阶段定义 — 不是这里。"
8. **部署/基础设施**（如适用）— 托管、CI/CD、环境
9. **开发计划** — 里程碑、任务分解、优先级排序
   - 必须定义带有明确退出标准的里程碑
   - 必须将特性分解为优先级任务（P0-P3）
   - 必须显示依赖链（Mermaid `graph`），识别关键路径
   - 必须包含带缓解策略的风险评估

展示每节。等待用户反馈。在继续下一节之前纳入更改。

**对于简单项目**（< 5 个特性）：将所有节合并为单一批准步骤，但仍包含所需的图表和依赖版本。

## 步骤 5：编写设计文档

保存批准的设计到 `docs/plans/YYYY-MM-DD-<topic>-design.md`。

### 模板使用

阅读步骤 2 中找到的模板（用户指定或默认 `docs/templates/design-template.md`）：
1. 保持模板的标题结构
2. 用批准的设计内容替换每个标题下的指导文本
3. 如果还没有，添加顶部元数据（`Date`、`Status`、`SRS Reference`、`Template` 路径）
4. 对于未覆盖的模板节：标记"[Not applicable]"
5. 对于没有匹配模板节的批准内容：附加为"Additional Notes"

## 步骤 5b：设计集成一致性检查

在过渡到 ATS 之前，机械验证跨特性集成一致性：

1. **契约完整性**：对于 §3.3 组件图中的每个边，验证 §6.2 内部 API 契约中存在对应的行。标记缺失行。
2. **模式一致性**：对于每个 §6.2 行，验证提供特性 的 §4.N 类图包含响应模式类型，且消费特性 的 §4.N 引用请求模式。标记不匹配。
3. **依赖完整性**：对于每个在 §6.2"Consumer"列中出现的特性，验证它在 §11.3 依赖链中列出提供特性。标记缺失的依赖边。

向用户呈现任何标记的问题。在继续到 ATS 之前解决。

## 步骤 6：过渡到 ATS

一旦设计文档保存并提交：

1. 总结 ATS 技能需要的关键输入：
   - **来自 SRS**：所有带验收标准的 FR/NFR/IFR 需求
   - **来自设计**：测试策略、技术栈、架构风险区域
2. **必需的子技能：** 调用 `long-task:long-task-ats` 生成验收测试策略

## 扩展设计阶段

| 项目规模 | 特性数量 | 设计深度 |
|---|---|---|
| 微型 | 1-5 | 单段方法 + 1 个批准步骤；逻辑视图 + 1 个特性图 + 依赖表 + 简化开发计划 |
| 小型 | 5-20 | 2-3 种方法选项 + 合并节批准；逻辑视图 + 关键特性图 + 依赖表 + 里程碑计划 |
| 中型 | 20-50 | 完整多节批准；所有架构视图 + 每个特性图 + 完整依赖分析 + 详细开发计划 |
| 大型 | 50-200+ | 完整多节批准；每个特性组的综合图表 + 依赖兼容性矩阵 + 带风险登记册的分阶段开发计划 |

## 第 4 节深度策略

对于具有许多特性的项目，§4.N 节以不同深度编写以管理上下文窗口约束：

| 项目规模 | 每个特性的 §4.N 内容 |
|---|---|
| 小型 (< 20) | 完整：概述 + 类图 + 行为图 + 设计笔记 + 集成表面 |
| 中型 (20-50) | P0/P1 特性完整；P2/P3 特性精简 |
| 大型 (50+) | 所有特性精简：概述 + 关键类型 + 集成表面 |

**精简 §4.N 格式：**

```markdown
### 4.N 特性：<名称> (FR-xxx)
#### 4.N.1 概述
[1-2 句话]
#### 4.N.2 关键类型
[列出此特性引入的主要类/类型，每个一带一行目的]
#### 4.N.6 集成表面
[引用 §6.2 的 Provides/Requires 表]
```

这是安全的，因为特性设计 SubAgent（Worker 步骤 4）产出完整的类/序列/流/算法设计，可访问 §6.2 契约。精简的 §4.N 作为**集成规范**，而不是完整设计。

## 红旗

| 合理化 | 正确回应 |
|---|---|
| "SRS 已经暗示了架构" | SRS 描述 WHAT，不是 HOW。展示选项。 |
| "只有一种方法构建这个" | 展示至少 2 种方法。即使明显的选择也受益于陈述的权衡。 |
| "我已经知道最好的方法" | 展示选项，让用户选择 |
| "用户似乎不耐烦，我跳过设计" | 简要解释价值，然后高效运行 |
| "我边做边设计" | 前置设计比会话中期更正更便宜 |
| "让我在这里重新澄清需求" | 需求属于 SRS。如果缺失，标记为开放问题并在设计前与用户解决。 |

## 图表要求

所有架构和设计视图**必须**使用 **Mermaid** 语法。这确保：
- 图表与文档一起版本控制（无外部图片文件）
- 图表可在 GitHub、GitLab 和大多数 Markdown 查看器中渲染
- 图表与设计更改保持同步

### 所需图表类型

| 节 | 图表类型 | Mermaid 语法 | 必需？ |
|---|---|---|---|
| 架构逻辑视图 | 分层包图 | `graph TB` | 始终 |
| 架构组件 | 组件交互 | `graph LR` | 始终 |
| 关键特性 — 结构 | 类图 | `classDiagram` | 每个特性 |
| 关键特性 — 行为 | 序列图 | `sequenceDiagram` | 每个特性（至少一个行为） |
| 关键特性 — 逻辑 | 流程/决策图 | `flowchart TD` | 每个特性（至少一个行为） |
| 数据模型 | ER 图 | `erDiagram` | 如果持久存储 |
| 依赖图 | 依赖树 | `graph LR` | 如果 > 3 个第三方依赖 |
| 开发计划 | 关键路径 | `graph LR` | 始终 |

### 图表质量检查清单

- [ ] 每个图表都有清晰的标题或周围标题
- [ ] 类图显示可见性修饰符（`+`/`-`/`#`）和关键方法
- [ ] 序列图显示主要成功路径和至少一个错误路径
- [ ] 流程图包含所有分支逻辑的决策节点
- [ ] 没有占位符图表 — 每个图表反映实际批准的设计内容
- [ ] §3.3 组件图中的每个边包含引用 §6.2 的契约 ID

## 第三方依赖规则

1. **需要确切版本** — 指定 `1.2.3` 或约束范围 `^1.2.0` / `>=1.2,<2.0`；永远不要使用 `latest` 或省略版本
2. **兼容性矩阵** — 验证每个依赖与以下兼容：
   - 目标语言/运行时版本（例如，Python >= 3.10，Node >= 18）
   - 堆栈中的其他依赖（检查已知冲突）
3. **许可证审计** — 记录每个依赖的许可证；标记可能与项目需求冲突的任何 copyleft 许可证（GPL、AGPL）
4. **升级路径** — 注意任何接近 EOL 或有已知迁移问题的依赖

## 开发计划规则

开发计划节桥接设计文档到 Init 阶段。它**必须**包含：

1. **里程碑** — 带明确范围和退出标准的时间盒阶段
   - M1 始终是"Foundation"（项目骨架、CI、核心抽象）
   - 最终里程碑始终是"Polish & Release"（NFR 验证、文档、示例）
2. **任务分解** — 映射到优先级（P0-P3）的特性，带理由
   - P0：Foundation — 所有其他特性所必需的
   - P1：核心价值 — 最小可行特性集
   - P2：扩展 — 重要但不是发布阻塞
   - P3：最好有 — 如果时间紧张则推迟
3. **依赖链** — Mermaid 图显示哪些特性阻塞其他
4. **配对特性排序** — 当项目同时有后端和前端特性时，组织任务分解表，使每个后端特性与其对应的前端特性配对。这产生自然开发流：后端 A → 前端 A → 后端 B → 前端 B。Init 阶段使用此配对在 `feature-list.json` 中排序特性。
5. **风险登记册** — 带缓解的技术和进度风险

Init 阶段使用此计划填充具有正确优先级排序、配对分组和依赖链的 `feature-list.json`。

## 集成

**被调用：** using-long-task（当 SRS + UCD 存在，没有设计文档，没有 feature-list.json）或 long-task-ucd（步骤 8）
**需要：** `docs/plans/*-srs.md` 处的批准 SRS；可选 `docs/plans/*-ucd.md` 处的批准 UCD（对于 UI 项目）
**链到：** long-task-ats（设计批准后）
**产出：** `docs/plans/YYYY-MM-DD-<topic>-design.md`
