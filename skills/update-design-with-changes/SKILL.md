# 根据变更更新模块设计文档

## 概述

此技能根据新的需求和 git 提交历史来更新现有的模块设计文档。它跟踪变更并维护版本历史表格。

## 使用场景

- 添加了新需求
- 现有功能被修改
- 需要记录需求变更
- 执行定期文档更新

## 工作流程

### 第一步：收集用户输入

1. **现有模块设计文档**：请用户提供：
   - 要更新的当前模块设计文档
   - 必须是 Markdown 格式

2. **新需求文档**：请用户提供：
   - 新的需求设计文档
   - 支持格式：PDF、DOCX、Markdown
   - 包含要应用的变更

3. **Git 提交历史**：请用户指定如何筛选提交：
   - **方式 A**：按分支名称（如 `feature/xxx`）
   - **方式 B**：按日期范围（如上个月）
   - **方式 C**：按特定提交（提供提交哈希）
   - **方式 D**：所有提交

4. **版本标识**：请用户指定此更新的版本：
   - 格式：v1.0、v1.1、v2.0 等
   - 或让系统自动生成

5. **输出**：请用户指定保存更新文档的位置
   - 默认为覆盖现有文件
   - 或保存为新文件

### 第二步：解析新需求

1. **提取新需求**
   - 解析新需求文档
   - 识别变更内容：新增、修改或删除
   - 如需要使用 pdf 处理或 docx 处理技能

2. **分析 Git 提交**
   - 根据用户的筛选条件获取提交
   - 提取提交信息
   - 提取变更文件
   - 总结技术变更

### 第三步：识别变更

将原始文档与新需求进行对比：

1. **新增内容**
   - 要添加的新功能
   - 新接口
   - 新数据模型

2. **修改内容**
   - 变更的业务逻辑
   - 更新的接口
   - 修改的数据结构

3. **删除内容**
   - 废弃的功能
   - 删除的接口

### 第四步：更新文档

1. **替换原始内容**
   - 更新受影响的章节
   - 保留未变更的章节

2. **重新生成图表**
   - 如果流程变更则更新流程图
   - 如果结构变更则更新类图
   - 如果数据模型变更则更新 ERD

3. **添加版本历史条目**
   - 在文档末尾，向版本表格添加新行：

```markdown
## 版本变更记录

| 版本 | 日期 | 变更内容 | 变更人 | 关联需求 |
|------|------|----------|--------|----------|
| v1.0 | 2024-01-01 | 初始版本 | - | REQ-001 |
| v1.1 | 2024-02-01 | 新增用户登出功能 | - | REQ-005 |
```

### 第五步：模板自我增强

更新文档后，询问用户：

> "是否启用模板自我增强？这将帮助改进更新流程。"

如果用户同意：
1. 展示更新的部分供用户审阅
2. 请用户确认变更
3. 应用反馈以改进未来更新

## 输出格式

更新后的模块设计文档保持与原始相同的结构，包含以下新增内容：

```markdown
# 模块名称 (已更新)

## [所有原始章节根据新需求更新]

...

## 版本变更记录

| 版本 | 日期 | 变更内容 | 变更人 | 关联需求/提交 |
|------|------|----------|--------|---------------|
| v1.0 | 2024-01-01 | 初始版本 | - | REQ-001 |
| v1.1 | 2024-02-01 | 新增用户登出功能，更新权限验证流程 | - | REQ-005, abc1234 |
| v1.2 | 2024-03-01 | 优化查询性能，添加缓存机制 | - | def5678 |
```

## Git 提交分析示例

分析 git 提交时，按以下方式总结变更：

| 提交 | 变更类型 | 变更描述 |
|------|----------|----------|
| abc1234 | 功能新增 | 新增用户登出接口 |
| def5678 | 性能优化 | 添加Redis缓存 |
| ghi9012 | Bug修复 | 修复权限验证漏洞 |

## 关键原则

1. **跟踪所有变更** - 每次更新都被记录
2. **维护版本历史** - 清晰的审计跟踪
3. **关联需求** - 跟踪哪个需求推动了每个变更
4. **关联提交** - 跟踪哪些代码提交实现了变更
5. **重新生成受影响的图表** - 保持图表同步
6. **请求用户确认** - 验证关键变更
7. **支持自我增强** - 改进更新流程
